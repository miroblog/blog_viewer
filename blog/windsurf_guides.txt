<h2>Cascade (Mode)</h2>
<p>Cascade는 Windsurf의 에이전트 기능의 핵심 축으로, 코드베이스에 대한 깊은 이해, 고급 도구 및 실시간 인식을 결합합니다.</p>
<p>작동 모드 (cmd + .  로 모드 변환)</p>
<ul>
<li><strong>Write 모드</strong>: AutoGPT와 유사하게, 최소한의 개입으로 여러 파일을 생성하고, 스크립트 실행, 테스트 수행, 코드 디버깅이 가능합니다.</li>
<li><strong>Chat 모드</strong>: 더 대화형으로, 프롬프트를 기반으로 코드 스니펫 생성 및 지침 제공에 중점을 둡니다.</li>
<li><strong>Legacy 모드</strong>: 현재 프로젝트와 관련 없는 일반 질문에 대해 ChatGPT처럼 기능합니다.</li>
</ul>

<h3>Codes, Files, Directories</h3>
<ul>
<li>참조해야할 범위를 좁혀 전달할때 더 좋은 결과를 얻기도함.</li>
<li>(주의) index되어 있어야 참조가 가능함 : <strong>Check Indexing Status:</strong> "Chat" 패널의 "Context" 창에서 녹색 점으로 indexing 상태를 확인할 수 있습니다.
<ul>
<li>
<p>세부사항</p>
<p>요약</p>
<ul>
<li>Windsurf는 코드베이스 전체를 인식하기 위해 로컬 indexing 엔진을 사용합니다.</li>
<li><strong>Automatic Indexing:</strong> 기본적으로 활성화되어 있습니다. (VS Code/JetBrains 확장과 다름)</li>
<li><strong>Embedding Generation:</strong> 코드를 이해하기 위해 embeddings을 생성합니다. 이를 통해 자연어 또는 코드 조각으로 검색할 수 있습니다.</li>
<li><strong>Local Storage:</strong> 코드 조각은 embedding 생성을 위해 원격 서버로 보내지만, 코드나 embedding은 원격으로 저장되지 않고 기기에 저장됩니다.</li>
<li>최초 indexing 시 CPU 자원을 소모하지만, 한 번만 발생하며 5-10분 정도 소요됩니다.</li>
<li><strong>Resource Requirements:</strong> 5,000개 파일 작업 공간의 경우 약 300MB의 RAM이 필요합니다.</li>
</ul>
<p><strong>Indexed Files 확인:</strong></p>
<ul>
<li>직접적으로 indexing된 파일 목록을 볼 수는 없습니다.</li>
<li><strong>Check Indexing Status:</strong> "Chat" 패널의 "Context" 창에서 녹색 점으로 indexing 상태를 확인할 수 있습니다.</li>
<li><strong>Excluded Files:</strong> 기본적으로 <strong><code>.gitignore</code></strong>, <strong><code>node_modules</code></strong>, 숨김 경로 (<strong><code>.</code></strong>)에 지정된 파일/폴더 및 <strong><code>.codeiumignore</code></strong> 파일에 지정된 파일/폴더는 indexing되지 않습니다.</li>
<li><strong>.codeiumignore:</strong> **<code>.gitignore</code>**와 동일한 문법을 사용하여 indexing에서 제외할 파일을 추가적으로 설정할 수 있습니다. repo root에 위치해야 합니다.</li>
<li><strong>Workspace Size Limits:</strong> "Max Workspace Size (File Count)" 설정으로 indexing할 최대 파일 수를 제한할 수 있습니다. (RAM 10GB 사용자 기준 10,000개 이하 권장)</li>
</ul>
<p><strong>주의할 점:</strong></p>
<ul>
<li>Indexing된 파일 목록을 직접 볼 수 없다는 점. Excluded Files에 주의하여 필요한 파일이 indexing에서 제외되지 않았는지 확인해야 합니다.</li>
<li><strong><code>.codeiumignore</code></strong> 파일을 활용하여 indexing 대상 파일을 세밀하게 조정할 수 있다는 점.</li>
<li>Workspace 크기가 커질수록 더 많은 자원을 소모하므로 "Max Workspace Size (File Count)" 설정을 적절하게 조정해야 한다는 점.</li>
</ul>
</li>
</ul>
</li>
</ul>

<h2>WindsurfTab</h2>
<ul>
<li>WindsurfTab은 코드 작성 워크플로우를 향상시키기 위한 기능입니다. autocomplete, supercomplete, tab to jump, tab to import 기능을 통합된 UX(tab)으로 제공합니다. 별도의 모델을 사용하여 코드 문맥을 파악하고 관련된 제안, 점프 및 임포트를 제공합니다.</li>
<li><strong>문맥 활용 :</strong> WindsurfTab은 더 나은 자동 완성을 위해 다음과 같은 요소가 고려됩니다.
<ul>
<li><strong>최근에 본 파일 (Recently Viewed Files):</strong> 다른 파일에서 작업한 코드가 제안에 영향을 줍니다.</li>
<li><strong>터미널 명령어 및 출력 (Terminal Commands &#x26; Outputs):</strong> 터미널에서 실행한 명령어와 그 결과가 의도를 파악하는 데 사용됩니다 (예: 임포트 실패 후 패키지 설치 제안).</li>
<li><strong>Cascade 대화 (Cascade Conversations):</strong> Cascade에서의 이전 상호 작용이 문맥을 유지하고 관련 연속을 제공하는 데 사용됩니다 (해당되는 경우).</li>
</ul>
</li>
</ul>
<p><img src="/windsurf_assets/image%204.png" alt="image.png"></p>

<h3>Supercomplete</h3>
<p>Supercomplete는 Windsurf의 고급 AI 기반 기능으로, 단순한 자동 완성(Autocomplete)을 넘어 사용자의 의도를 예측합니다. 단순히 다음 줄의 코드를 제안하는 것이 아니라 의도를 파악합니다. 이는 Windsurf Tab 시스템의 일부로서, 단 하나의 키 입력으로 다양한 코드 변경을 가능하게 합니다.</p>

<h3>Tab to Import</h3>
<p>Tab to Import는 파일에 새로운 의존성(dependency)을 정의한 후, 힌트가 표시될 때**<code>tab</code>**키를 누르면 해당 의존성을 파일 상단에 자동으로 import하는 기능입니다. 커서는 현재 위치에 그대로 유지됩니다.</p>
<ul>
<li>파일에서 새로운 라이브러리나 모듈을 사용합니다</li>
<li>Windsurf가 import 힌트를 표시합니다</li>
<li>**<code>tab</code>**키를 누르면 자동으로 파일 상단에 import 구문이 추가됩니다</li>
<li>작업 중인 위치에서 커서가 이동하지 않고 계속 작업할 수 있습니다</li>
</ul>
<h3>Tab to Jump</h3>
<ul>
<li><strong>사용법:</strong> 키워드 (예: 함수 이름, 변수 이름)를 입력하고 <strong>Tab</strong> 키를 누릅니다. Windsurf Tab이 점프 가능한 위치를 감지하면 점프 옵션이 있는 위젯을 표시합니다.</li>
<li><strong>(v 1.5.6)</strong> 점프 거리가 두 배 이상 늘어나 코드베이스 내에서 더 멀리 이동할 수 있습니다. 점프 위젯의 UI가 새로워졌습니다.</li>
<li>Tab to Jump 위젯을 <em>클릭</em>하여 원하는 점프 위치를 선택할 수 있습니다.</li>
</ul>

<h2>메모리 (Memory)</h2>
<h3><strong>자동 메모리 (Automated Memories)</strong></h3>
<ul>
<li><strong>목표</strong>: Cascade와 대화할 때 중요한 내용을 자동으로 기억해 줍니다.</li>
<li><strong>만드는 방법</strong>:
<ul>
<li>Cascade와 대화하는 동안 자동으로 생성됩니다.</li>
<li>"create a memory of..."와 같이 Cascade에게 직접 요청해서 만들 수도 있습니다.</li>
</ul>
</li>
<li><strong>저장 위치</strong>: 각 워크스페이스(workspace)에 연결되어 저장됩니다.</li>
<li><strong>불러오기</strong>: 현재 상황에 맞는 정보가 자동으로 나타납니다.</li>
<li><strong>비용</strong>: 크레딧(credits)을 사용하지 않습니다.</li>
<li><strong>이럴 때 좋아요</strong>:
<ul>
<li>프로젝트(project)를 진행하면서 나오는 중요한 내용</li>
<li>사용자(user)의 선호도나 대화 중에 결정된 사항</li>
<li>중요한 코드 스니펫(code snippets)이나 아키텍처 관련 결정</li>
<li>여러 세션(session)에서 반복적으로 필요할 수 있는 정보</li>
</ul>
</li>
</ul>
<h3><strong>명시적 메모리 (Explicit Memory)</strong></h3>
<ul>
<li><strong>종류</strong>:
<ol>

<li><strong>로컬 규칙 (Local Rules)</strong> (<code>.windsurfrules</code>): 특정 워크스페이스에만 적용됩니다.</li>
</ol>
</li>
<li><strong>제한 사항</strong>: 파일 하나당 6,000자 (총 12,000자)까지 쓸 수 있습니다.</li>
<li><strong>관리 방법</strong>: Windsurf Settings > Edit Rules에서 수정할 수 있습니다.</li>
<li><strong>이럴 때 좋아요</strong>:
<ul>
<li>프로젝트 전반에 걸쳐 지켜야 할 규칙과 표준</li>
<li>전체 프로젝트에 적용되는 코딩 컨벤션(coding conventions)</li>
<li>팀(team)에서 정한 규칙이나 요구 사항</li>
<li>Document 설정이나 코딩 스타일 관련 선호도</li>
</ul>
</li>
</ul>

<h2>(Advanced) 디렉토리별 메모리, Rule 관리 전략</h2>
<h3>Prompt Context란 ?</h3>
<p>Prompt Context 접근 방식은 코드 저장소 내에서 작업하는 AI 어시스턴트에게 디렉토리별 가이드를 제공해 어떤 컨텍스트와 룰을 우선시 할 것인지에 대한 관리를 좀 더 쉽게 할 수 있는 방법론입니다.</p>
<h3>설정 방법</h3>
<ul>
<li><a href="https://gist.github.com/miroblog/dc5e946e0437a4b054b78d8f166af537">Prompt Context Rules</a></li>
</ul>
<ol>
<li>위 rule을 .windsurfrules에 명시합니다.</li>
<li>특수 지시사항이 필요한 하위 디렉토리에 디렉토리별 컨텍스트(.context.md), 규칙(.AI.md) 파일을추가합니다</li>
</ol>
<p>특정 디렉토리 내에서의 작업 X(코드 수정 및 생성 등)에 필요한 규칙(formatting rule 등) 과 컨텍스트(예시 snippet 등) 을 개발자가 효과적으로 통제할 수 있습니다.</p>
<h2>(Advanced) Memory Bank를 통한 관리 전략</h2>
<h3><strong>Memory Bank란 무엇인가?</strong></h3>
<p>Memory Bank는 Windsurf AI가 여러 세션에 걸쳐 프로젝트 컨텍스트를 유지할 수 있게 해주는 구조화된 문서화 시스템입니다. AI의 "메모리" 역할을 하는 마크다운 파일들을 생성하고 업데이트하여 더 일관되고 효율적인 지원을 가능하게 합니다.</p>
<ul>
<li>세션 간에 프로젝트 컨텍스트 유지</li>
<li>프로젝트 세부 정보 자동 문서화</li>
<li>AI가 프로젝트 패턴과 제약 조건을 이해하도록 보장</li>
<li>반복적인 설명 감소</li>
<li>코드 일관성 향상</li>
</ul>
<h3><strong>설정 방법</strong></h3>
<p><strong>1단계: Memory Bank 규칙 생성</strong></p>
<ul>
<li><a href="https://gist.github.com/miroblog/7a60b45b572dc4922aac21de57a0915f">Memory Bank Rules</a></li>
</ul>
<ol>
<li>Memory Bank 규칙을 복사</li>
<li>windsurf 설정 → Rules로 이동</li>
<li>.windsurfrules 섹션에 규칙을 붙여넣기</li>
<li>설정 저장</li>
</ol>
<p><strong>2단계: Memory Bank 초기화</strong></p>
<pre><code>1. windsurf AI와 새 채팅 시작
2. 다음과 같이 입력: `initialize memory bank`
3. windsurf가 memory bank 폴더 구조를 생성하도록 함
</code></pre>
<h3><strong>Memory Bank 구조</strong></h3>
<pre><code>1. **projectbrief.md** - 기초 문서
    - 핵심 요구사항 및 목표
    - 프로젝트 범위
2. **productContext.md** - 제품 세부 정보
    - 이 프로젝트가 존재하는 이유
    - 해결하는 문제점
    - 사용자 경험 목표
3. **systemPatterns.md** - 기술 아키텍처
    - 시스템 아키텍처
    - 주요 디자인 패턴
    - 컴포넌트 관계
4. **techContext.md** - 기술적 세부 사항
    - 사용된 기술
    - 개발 설정
    - 의존성
5. **activeContext.md** - 현재 작업
    - 최근 변경 사항
    - 다음 단계
    - 활성 의사 결정
6. **progress.md** - 프로젝트 상태
    - 잘 작동하는 것
    - 구축해야 할 것
    - 알려진 문제점
</code></pre>
<h3><strong>Memory Bank 사용 방법</strong></h3>
<p><strong>Plan Mode</strong></p>
<pre><code>구현 전에 windsurf가 기능 계획을 도와주길 원할 때:

1. 새 채팅 시작
2. 다음과 같이 입력: `use plan mode`
3. 계획하고 싶은 내용 설명
4. windsurf가 memory bank 파일을 읽고 계획을 도와줌

예시:

use plan mode
Let's plan how to implement user authentication in our app


</code></pre>
<p><strong>Implementation Mode</strong></p>
<pre><code>기능을 구현하고 싶을 때:

1. 구현하고 싶은 내용 설명
2. windsurf가 memory bank를 읽고 프로젝트 패턴에 따라 구현
</code></pre>
<p><strong>Memory Bank 업데이트</strong></p>
<pre><code>Memory Bank는 windsurf와 작업할 때 자동으로 업데이트되지만, 수동으로 업데이트를 트리거할 수 있습니다:

1. 다음과 같이 입력: `update memory bank`
2. windsurf가 모든 파일을 검토하고 문서를 업데이트
</code></pre>
<p><strong>사용팁</strong></p>
<pre><code>- 작업 설명 시 구체적으로 설명
- 정확성을 보장하기 위해 memory bank 파일을 가끔 검토
- 새 프로젝트의 경우, 초기화할 때 초기 컨텍스트 제공
- 복잡한 기능은 구현 전에 plan mode 사용
- 중요한 프로젝트 변경 후 memory bank 업데이트
- windsurf가 memory bank를 무시하는 것 같다면, "use plan mode"를 명시적으로 지정
- 구현이 프로젝트 패턴과 일치하지 않는 경우, systemPatterns.md 파일 확인
- 대규모 프로젝트의 경우, memory-bank 폴더에 추가 컨텍스트 파일 생성 고려
</code></pre>
<h1>MCP 서버 관련 기능</h1>
<p>Model Context Protocol(MCP)은 개발 환경에서 직접 외부 서비스 및 API에 연결할 수 있도록 Windsurf의 기능을 확장하는 기능입니다.</p>

<h1>AI 개발 방법론 (case by case 보충 필요)</h1>
<h2>Meta Prompting</h2>
<p>메타 프롬프팅은 직접적인 지시 (일일이 모든 단계를 직접 나열하는 대신) AI에게 포괄적인 계획 수립을 요청하는 방식입니다. 이를 통해 AI가 복잡한 작업을 체계화하고 가이드하는 방식입니다.  예를 들어, "Tech stack" 선택이나 "Authentication" 처리 같은 작업을 직접 적는 대신, "Slack 봇을 만들려고 하는데, 봇 설정에 필요한 단계별 계획을 세울 때 Best Practice와 잠재적인 문제점을 고려해서 가이드해 줄 수 있어?" 와 같이 AI에게 물어볼 수 있습니다.</p>
<h2>Plan → Implement → Feedback → … → Update Memory</h2>
<ol>
<li>먼저 계획(Plan) 단계에서는 무엇을 만들지 명확히 정의하고, 현재 코드 구조를 분석한 후, AI가 단계별 접근 방식을 제안하며 테스트 케이스를 먼저 작성하도록 지시 .</li>
<li>구현(Implement) 단계에서는 AI가 계획에 따라 코드를 생성하고, 기능 검증을 위한 테스트를 작성하며, 여러 파일을 한 번에 수정하고 필요한 명령어를 실행</li>
<li>피드백(Feedback) 단계에서는 테스트를 실행하여 요구사항 충족 여부를 확인하고, 오류와 문제점을 찾아내며, 코드 성능을 확인하고 코드 품질을 검토</li>
<li>개선 구현(Implement) 단계에서는 발견된 오류를 수정하고, 코드를 정리하고 개선하며, 성능을 최적화하고 코드에 설명을 추가</li>
<li>마지막으로 메모리 업데이트(Update Memory) 단계에서는 중요한 결정사항을 별도의 메모리(.md)에 기록하고, 반복되는 패턴을 식별하며, 프로젝트 지식을 축적하고 개발자의 코딩 스타일을 이해하도록 유도</li>
</ol>

<h1>상세 설정</h1>
<h2>Cascade 설정</h2>
<h3>Auto-Fix Lints</h3>
<ul>
<li><strong>설정</strong>: On/Off</li>
<li><strong>기능</strong>: 활성화되면 Cascade가 생성된 코드의 린팅 오류를 자동으로 수정합니다. 비활성화되면 자동 수정 없이 코드가 생성됩니다.</li>
</ul>
<h3>모델 선택</h3>
<p>작업 복잡성 및 예산에 따라 다양한 AI 모델 중에서 선택:</p>
<ul>
<li><strong>Gemini Flash</strong>: 간단한 작업에 적합, 낮은 비용</li>
<li><strong>GPT-4o / Claude 3.5 / Claude 3.7</strong>: 유사한 비용, 높은 품질</li>
<li><strong>Claude 3.7 Thinking</strong>: 최고 품질, 최고 비용</li>
</ul>
<p>모델 선택 고려 사항:</p>
<ul>
<li>작업 복잡성 요구 사항</li>
<li>속도 vs 품질 트레이드오프</li>
<li>크레딧 소비율</li>
</ul>
<h2>터미널 설정</h2>
<h3>터미널 자동 실행(Turbo Mode)</h3>
<ul>
<li><strong>설정</strong>: On/Off</li>
<li><strong>기능</strong>: 활성화되면 Cascade가 허용/차단 목록 규칙에 따라 수동 승인 없이 터미널 명령을 실행할 수 있습니다. 비활성화되면 모든 명령에 승인이 필요합니다.</li>
</ul>
<h3>명령 허용/차단 목록</h3>
<p>자동으로 실행할 수 있는 명령 구성:</p>
<ul>
<li><strong>허용 목록</strong>: 항상 실행할 수 있는 명령 지정</li>
<li><strong>차단 목록</strong>: 자동으로 실행하면 안 되는 명령 나열</li>
<li>git 명령 허용: <code>git*</code></li>
<li>파일 삭제 차단: <code>rm -rf*</code></li>
</ul>
<h2>Auto Complete 설정</h2>
<h3>자동완성</h3>
<ul>
<li><strong>설정</strong>: On/Off</li>
<li><strong>기능</strong>: 기본 코드 완성 기능을 켜거나 끕니다.</li>
</ul>
<h3>자동완성 속도</h3>
<ul>
<li><strong>설정</strong>: Fast/Normal</li>
<li><strong>기능</strong>: 더 빠른 제안(Fast) 또는 더 신중한 타이밍(Normal) 중에서 선택합니다.</li>
</ul>
<h2>Supercomplete</h2>
<ul>
<li><strong>설정</strong>: On/Off</li>
<li><strong>기능</strong>: 더 포괄적인 제안을 제공하는 향상된 컨텍스트 인식 코드 완성 기능을 켜거나 끕니다.</li>
</ul>
<h2>네비게이션 설정</h2>
<h3>Tab to Jump</h3>
<ul>
<li><strong>설정</strong>: On/Off</li>
<li><strong>기능</strong>: 활성화되면 Tab을 사용하여 import 문, 종속성 및 관련 코드 섹션으로 이동할 수 있습니다.</li>
</ul>
<h2>메모리 및 규칙 구성</h2>

<ul>

<li>가장 중요한 규칙 우선시</li>
<li>코딩 표준 및 패턴에 대해 구체적으로 명시</li>

<h2>Cascade (Mode)</h2>
<p>Cascade는 Windsurf의 에이전트 기능의 핵심 축으로, 코드베이스에 대한 깊은 이해, 고급 도구 및 실시간 인식을 결합합니다.</p>
<p>작동 모드 (cmd + .  로 모드 변환)</p>
<ul>
<li><strong>Write 모드</strong>: AutoGPT와 유사하게, 최소한의 개입으로 여러 파일을 생성하고, 스크립트 실행, 테스트 수행, 코드 디버깅이 가능합니다.</li>
<li><strong>Chat 모드</strong>: 더 대화형으로, 프롬프트를 기반으로 코드 스니펫 생성 및 지침 제공에 중점을 둡니다.</li>
<li><strong>Legacy 모드</strong>: 현재 프로젝트와 관련 없는 일반 질문에 대해 ChatGPT처럼 기능합니다.</li>
</ul>

<h3>Codes, Files, Directories</h3>
<ul>
<li>참조해야할 범위를 좁혀 전달할때 더 좋은 결과를 얻기도함.</li>
<li>(주의) index되어 있어야 참조가 가능함 : <strong>Check Indexing Status:</strong> "Chat" 패널의 "Context" 창에서 녹색 점으로 indexing 상태를 확인할 수 있습니다.
<ul>
<li>
<p>세부사항</p>
<p>요약</p>
<ul>
<li>Windsurf는 코드베이스 전체를 인식하기 위해 로컬 indexing 엔진을 사용합니다.</li>
<li><strong>Automatic Indexing:</strong> 기본적으로 활성화되어 있습니다. (VS Code/JetBrains 확장과 다름)</li>
<li><strong>Embedding Generation:</strong> 코드를 이해하기 위해 embeddings을 생성합니다. 이를 통해 자연어 또는 코드 조각으로 검색할 수 있습니다.</li>
<li><strong>Local Storage:</strong> 코드 조각은 embedding 생성을 위해 원격 서버로 보내지만, 코드나 embedding은 원격으로 저장되지 않고 기기에 저장됩니다.</li>
<li>최초 indexing 시 CPU 자원을 소모하지만, 한 번만 발생하며 5-10분 정도 소요됩니다.</li>
<li><strong>Resource Requirements:</strong> 5,000개 파일 작업 공간의 경우 약 300MB의 RAM이 필요합니다.</li>
</ul>
<p><strong>Indexed Files 확인:</strong></p>
<ul>
<li>직접적으로 indexing된 파일 목록을 볼 수는 없습니다.</li>
<li><strong>Check Indexing Status:</strong> "Chat" 패널의 "Context" 창에서 녹색 점으로 indexing 상태를 확인할 수 있습니다.</li>
<li><strong>Excluded Files:</strong> 기본적으로 <strong><code>.gitignore</code></strong>, <strong><code>node_modules</code></strong>, 숨김 경로 (<strong><code>.</code></strong>)에 지정된 파일/폴더 및 <strong><code>.codeiumignore</code></strong> 파일에 지정된 파일/폴더는 indexing되지 않습니다.</li>
<li><strong>.codeiumignore:</strong> **<code>.gitignore</code>**와 동일한 문법을 사용하여 indexing에서 제외할 파일을 추가적으로 설정할 수 있습니다. repo root에 위치해야 합니다.</li>
<li><strong>Workspace Size Limits:</strong> "Max Workspace Size (File Count)" 설정으로 indexing할 최대 파일 수를 제한할 수 있습니다. (RAM 10GB 사용자 기준 10,000개 이하 권장)</li>
</ul>
<p><strong>주의할 점:</strong></p>
<ul>
<li>Indexing된 파일 목록을 직접 볼 수 없다는 점. Excluded Files에 주의하여 필요한 파일이 indexing에서 제외되지 않았는지 확인해야 합니다.</li>
<li><strong><code>.codeiumignore</code></strong> 파일을 활용하여 indexing 대상 파일을 세밀하게 조정할 수 있다는 점.</li>
<li>Workspace 크기가 커질수록 더 많은 자원을 소모하므로 "Max Workspace Size (File Count)" 설정을 적절하게 조정해야 한다는 점.</li>
</ul>
</li>
</ul>
</li>
</ul>

<h2>WindsurfTab</h2>
<ul>
<li>WindsurfTab은 코드 작성 워크플로우를 향상시키기 위한 기능입니다. autocomplete, supercomplete, tab to jump, tab to import 기능을 통합된 UX(tab)으로 제공합니다. 별도의 모델을 사용하여 코드 문맥을 파악하고 관련된 제안, 점프 및 임포트를 제공합니다.</li>
<li><strong>문맥 활용 :</strong> WindsurfTab은 더 나은 자동 완성을 위해 다음과 같은 요소가 고려됩니다.
<ul>
<li><strong>최근에 본 파일 (Recently Viewed Files):</strong> 다른 파일에서 작업한 코드가 제안에 영향을 줍니다.</li>
<li><strong>터미널 명령어 및 출력 (Terminal Commands &#x26; Outputs):</strong> 터미널에서 실행한 명령어와 그 결과가 의도를 파악하는 데 사용됩니다 (예: 임포트 실패 후 패키지 설치 제안).</li>
<li><strong>Cascade 대화 (Cascade Conversations):</strong> Cascade에서의 이전 상호 작용이 문맥을 유지하고 관련 연속을 제공하는 데 사용됩니다 (해당되는 경우).</li>
</ul>
</li>
</ul>

<h3>Tab to Import</h3>
<p>Tab to Import는 파일에 새로운 의존성(dependency)을 정의한 후, 힌트가 표시될 때**<code>tab</code>**키를 누르면 해당 의존성을 파일 상단에 자동으로 import하는 기능입니다. 커서는 현재 위치에 그대로 유지됩니다.</p>
<ul>
<li>파일에서 새로운 라이브러리나 모듈을 사용합니다</li>
<li>Windsurf가 import 힌트를 표시합니다</li>
<li>**<code>tab</code>**키를 누르면 자동으로 파일 상단에 import 구문이 추가됩니다</li>
<li>작업 중인 위치에서 커서가 이동하지 않고 계속 작업할 수 있습니다</li>
</ul>
<h3>Tab to Jump</h3>
<ul>
<li><strong>사용법:</strong> 키워드 (예: 함수 이름, 변수 이름)를 입력하고 <strong>Tab</strong> 키를 누릅니다. Windsurf Tab이 점프 가능한 위치를 감지하면 점프 옵션이 있는 위젯을 표시합니다.</li>
<li><strong>(v 1.5.6)</strong> 점프 거리가 두 배 이상 늘어나 코드베이스 내에서 더 멀리 이동할 수 있습니다. 점프 위젯의 UI가 새로워졌습니다.</li>
<li>Tab to Jump 위젯을 <em>클릭</em>하여 원하는 점프 위치를 선택할 수 있습니다.</li>
</ul>

<h2>메모리 (Memory)</h2>
<h3><strong>자동 메모리 (Automated Memories)</strong></h3>
<ul>
<li><strong>목표</strong>: Cascade와 대화할 때 중요한 내용을 자동으로 기억해 줍니다.</li>
<li><strong>만드는 방법</strong>:
<ul>
<li>Cascade와 대화하는 동안 자동으로 생성됩니다.</li>
<li>"create a memory of..."와 같이 Cascade에게 직접 요청해서 만들 수도 있습니다.</li>
</ul>
</li>
<li><strong>저장 위치</strong>: 각 워크스페이스(workspace)에 연결되어 저장됩니다.</li>
<li><strong>불러오기</strong>: 현재 상황에 맞는 정보가 자동으로 나타납니다.</li>
<li><strong>비용</strong>: 크레딧(credits)을 사용하지 않습니다.</li>
<li><strong>이럴 때 좋아요</strong>:
<ul>
<li>프로젝트(project)를 진행하면서 나오는 중요한 내용</li>
<li>사용자(user)의 선호도나 대화 중에 결정된 사항</li>
<li>중요한 코드 스니펫(code snippets)이나 아키텍처 관련 결정</li>
<li>여러 세션(session)에서 반복적으로 필요할 수 있는 정보</li>
</ul>
</li>
</ul>
<h3><strong>명시적 메모리 (Explicit Memory)</strong></h3>
<ul>
<li><strong>종류</strong>:
<ol>
<li><strong>글로벌 규칙 (Global Rules)</strong> (<code>global_rules.md</code>): 모든 워크스페이스에 공통으로 적용됩니다.</li>
<li><strong>로컬 규칙 (Local Rules)</strong> (<code>.windsurfrules</code>): 특정 워크스페이스에만 적용됩니다.</li>
</ol>
</li>
<li><strong>제한 사항</strong>: 파일 하나당 6,000자 (총 12,000자)까지 쓸 수 있습니다.</li>
<li><strong>관리 방법</strong>: Windsurf Settings > Edit Rules에서 수정할 수 있습니다.</li>
<li><strong>이럴 때 좋아요</strong>:
<ul>
<li>프로젝트 전반에 걸쳐 지켜야 할 규칙과 표준</li>
<li>전체 프로젝트에 적용되는 코딩 컨벤션(coding conventions)</li>
<li>팀(team)에서 정한 규칙이나 요구 사항</li>
<li>Document 설정이나 코딩 스타일 관련 선호도</li>
</ul>
</li>
</ul>

<h2>(Advanced) 디렉토리별 메모리, Rule 관리 전략</h2>
<h3>Prompt Context란 ?</h3>
<p>Prompt Context 접근 방식은 코드 저장소 내에서 작업하는 AI 어시스턴트에게 디렉토리별 가이드를 제공해 어떤 컨텍스트와 룰을 우선시 할 것인지에 대한 관리를 좀 더 쉽게 할 수 있는 방법론입니다.</p>
<h3>설정 방법</h3>
<ul>
<li><a href="https://gist.github.com/miroblog/dc5e946e0437a4b054b78d8f166af537">Prompt Context Rules</a></li>
</ul>
<ol>
<li>위 rule을 .windsurfrules에 명시합니다.</li>
<li>특수 지시사항이 필요한 하위 디렉토리에 디렉토리별 컨텍스트(.context.md), 규칙(.AI.md) 파일을추가합니다</li>
</ol>
<p>특정 디렉토리 내에서의 작업 X(코드 수정 및 생성 등)에 필요한 규칙(formatting rule 등) 과 컨텍스트(예시 snippet 등) 을 개발자가 효과적으로 통제할 수 있습니다.</p>
<h2>(Advanced) Memory Bank를 통한 관리 전략</h2>
<h3><strong>Memory Bank란 무엇인가?</strong></h3>
<p>Memory Bank는 Windsurf AI가 여러 세션에 걸쳐 프로젝트 컨텍스트를 유지할 수 있게 해주는 구조화된 문서화 시스템입니다. AI의 "메모리" 역할을 하는 마크다운 파일들을 생성하고 업데이트하여 더 일관되고 효율적인 지원을 가능하게 합니다.</p>
<ul>
<li>세션 간에 프로젝트 컨텍스트 유지</li>
<li>프로젝트 세부 정보 자동 문서화</li>
<li>AI가 프로젝트 패턴과 제약 조건을 이해하도록 보장</li>
<li>반복적인 설명 감소</li>
<li>코드 일관성 향상</li>
</ul>
<h3><strong>설정 방법</strong></h3>
<p><strong>1단계: Memory Bank 규칙 생성</strong></p>
<ul>
<li><a href="https://gist.github.com/miroblog/7a60b45b572dc4922aac21de57a0915f">Memory Bank Rules</a></li>
</ul>
<ol>
<li>Memory Bank 규칙을 복사</li>
<li>windsurf 설정 → Rules로 이동</li>
<li>.windsurfrules 섹션에 규칙을 붙여넣기</li>
<li>설정 저장</li>
</ol>
<p><strong>2단계: Memory Bank 초기화</strong></p>
<pre><code>1. windsurf AI와 새 채팅 시작
2. 다음과 같이 입력: `initialize memory bank`
3. windsurf가 memory bank 폴더 구조를 생성하도록 함
</code></pre>
<h3><strong>Memory Bank 구조</strong></h3>
<pre><code>1. **projectbrief.md** - 기초 문서
    - 핵심 요구사항 및 목표
    - 프로젝트 범위
2. **productContext.md** - 제품 세부 정보
    - 이 프로젝트가 존재하는 이유
    - 해결하는 문제점
    - 사용자 경험 목표
3. **systemPatterns.md** - 기술 아키텍처
    - 시스템 아키텍처
    - 주요 디자인 패턴
    - 컴포넌트 관계
4. **techContext.md** - 기술적 세부 사항
    - 사용된 기술
    - 개발 설정
    - 의존성
5. **activeContext.md** - 현재 작업
    - 최근 변경 사항
    - 다음 단계
    - 활성 의사 결정
6. **progress.md** - 프로젝트 상태
    - 잘 작동하는 것
    - 구축해야 할 것
    - 알려진 문제점
</code></pre>
<h3><strong>Memory Bank 사용 방법</strong></h3>
<p><strong>Plan Mode</strong></p>
<pre><code>구현 전에 windsurf가 기능 계획을 도와주길 원할 때:

1. 새 채팅 시작
2. 다음과 같이 입력: `use plan mode`
3. 계획하고 싶은 내용 설명
4. windsurf가 memory bank 파일을 읽고 계획을 도와줌

예시:

use plan mode
Let's plan how to implement user authentication in our app


</code></pre>
<p><strong>Implementation Mode</strong></p>
<pre><code>기능을 구현하고 싶을 때:

1. 구현하고 싶은 내용 설명
2. windsurf가 memory bank를 읽고 프로젝트 패턴에 따라 구현
</code></pre>
<p><strong>Memory Bank 업데이트</strong></p>
<pre><code>Memory Bank는 windsurf와 작업할 때 자동으로 업데이트되지만, 수동으로 업데이트를 트리거할 수 있습니다:

1. 다음과 같이 입력: `update memory bank`
2. windsurf가 모든 파일을 검토하고 문서를 업데이트
</code></pre>
<p><strong>사용팁</strong></p>
<pre><code>- 작업 설명 시 구체적으로 설명
- 정확성을 보장하기 위해 memory bank 파일을 가끔 검토
- 새 프로젝트의 경우, 초기화할 때 초기 컨텍스트 제공
- 복잡한 기능은 구현 전에 plan mode 사용
- 중요한 프로젝트 변경 후 memory bank 업데이트
- windsurf가 memory bank를 무시하는 것 같다면, "use plan mode"를 명시적으로 지정
- 구현이 프로젝트 패턴과 일치하지 않는 경우, systemPatterns.md 파일 확인
- 대규모 프로젝트의 경우, memory-bank 폴더에 추가 컨텍스트 파일 생성 고려
</code></pre>

<h1>AI 개발 방법론 (case by case 보충 필요)</h1>
<h2>Meta Prompting</h2>
<p>메타 프롬프팅은 직접적인 지시 (일일이 모든 단계를 직접 나열하는 대신) AI에게 포괄적인 계획 수립을 요청하는 방식입니다. 이를 통해 AI가 복잡한 작업을 체계화하고 가이드하는 방식입니다.  예를 들어, "Tech stack" 선택이나 "Authentication" 처리 같은 작업을 직접 적는 대신, "Slack 봇을 만들려고 하는데, 봇 설정에 필요한 단계별 계획을 세울 때 Best Practice와 잠재적인 문제점을 고려해서 가이드해 줄 수 있어?" 와 같이 AI에게 물어볼 수 있습니다.</p>
<h2>Plan → Implement → Feedback → … → Update Memory</h2>
<ol>
<li>먼저 계획(Plan) 단계에서는 무엇을 만들지 명확히 정의하고, 현재 코드 구조를 분석한 후, AI가 단계별 접근 방식을 제안하며 테스트 케이스를 먼저 작성하도록 지시 .</li>
<li>구현(Implement) 단계에서는 AI가 계획에 따라 코드를 생성하고, 기능 검증을 위한 테스트를 작성하며, 여러 파일을 한 번에 수정하고 필요한 명령어를 실행</li>
<li>피드백(Feedback) 단계에서는 테스트를 실행하여 요구사항 충족 여부를 확인하고, 오류와 문제점을 찾아내며, 코드 성능을 확인하고 코드 품질을 검토</li>
<li>개선 구현(Implement) 단계에서는 발견된 오류를 수정하고, 코드를 정리하고 개선하며, 성능을 최적화하고 코드에 설명을 추가</li>
<li>마지막으로 메모리 업데이트(Update Memory) 단계에서는 중요한 결정사항을 별도의 메모리(.md)에 기록하고, 반복되는 패턴을 식별하며, 프로젝트 지식을 축적하고 개발자의 코딩 스타일을 이해하도록 유도</li>
</ol>

<h1>상세 설정</h1>
<h2>Cascade 설정</h2>
<h3>Auto-Fix Lints</h3>
<ul>
<li><strong>설정</strong>: On/Off</li>
<li><strong>기능</strong>: 활성화되면 Cascade가 생성된 코드의 린팅 오류를 자동으로 수정합니다. 비활성화되면 자동 수정 없이 코드가 생성됩니다.</li>
</ul>
<h3>모델 선택</h3>
<p>작업 복잡성 및 예산에 따라 다양한 AI 모델 중에서 선택:</p>
<ul>
<li><strong>Gemini Flash</strong>: 간단한 작업에 적합, 낮은 비용</li>
<li><strong>GPT-4o / Claude 3.5 / Claude 3.7</strong>: 유사한 비용, 높은 품질</li>
<li><strong>Claude 3.7 Thinking</strong>: 최고 품질, 최고 비용</li>
</ul>
<p>모델 선택 고려 사항:</p>
<ul>
<li>작업 복잡성 요구 사항</li>
<li>속도 vs 품질 트레이드오프</li>
<li>크레딧 소비율</li>
</ul>
<h2>터미널 설정</h2>
<h3>터미널 자동 실행(Turbo Mode)</h3>
<ul>
<li><strong>설정</strong>: On/Off</li>
<li><strong>기능</strong>: 활성화되면 Cascade가 허용/차단 목록 규칙에 따라 수동 승인 없이 터미널 명령을 실행할 수 있습니다. 비활성화되면 모든 명령에 승인이 필요합니다.</li>
</ul>
<h3>명령 허용/차단 목록</h3>
<p>자동으로 실행할 수 있는 명령 구성:</p>
<ul>
<li><strong>허용 목록</strong>: 항상 실행할 수 있는 명령 지정</li>
<li><strong>차단 목록</strong>: 자동으로 실행하면 안 되는 명령 나열</li>
<li>git 명령 허용: <code>git*</code></li>
<li>파일 삭제 차단: <code>rm -rf*</code></li>
</ul>
<h2>Auto Complete 설정</h2>
<h3>자동완성</h3>
<ul>
<li><strong>설정</strong>: On/Off</li>
<li><strong>기능</strong>: 기본 코드 완성 기능을 켜거나 끕니다.</li>
</ul>
<h3>자동완성 속도</h3>
<ul>
<li><strong>설정</strong>: Fast/Normal</li>
<li><strong>기능</strong>: 더 빠른 제안(Fast) 또는 더 신중한 타이밍(Normal) 중에서 선택합니다.</li>
</ul>
<h2>Supercomplete</h2>
<ul>
<li><strong>설정</strong>: On/Off</li>
<li><strong>기능</strong>: 더 포괄적인 제안을 제공하는 향상된 컨텍스트 인식 코드 완성 기능을 켜거나 끕니다.</li>
</ul>
<h2>네비게이션 설정</h2>
<h3>Tab to Jump</h3>
<ul>
<li><strong>설정</strong>: On/Off</li>
<li><strong>기능</strong>: 활성화되면 Tab을 사용하여 import 문, 종속성 및 관련 코드 섹션으로 이동할 수 있습니다.</li>
</ul>
<h2>메모리 및 규칙 구성</h2>
<h3>메모리 생성 및 관리</h3>
<p>다음을 통해 접근:</p>
<ul>
<li>"create memory…" 로 Cascade에 메모리 생성 요청</li>
<li>오른쪽 상단 "Manage Memory" 옵션을 통해 메모리 관리</li>
</ul>
<h3>규칙 유형</h3>
<ul>
<li><strong>전역 규칙</strong>: 모든 작업 공간에 적용</li>
<li><strong>로컬 규칙</strong>(windsurfrules): 특정 프로젝트에 적용
<ul>
<li>프로젝트의 <code>.windsurfrules</code> 파일에 저장</li>
<li>12,000자로 제한(전역 규칙과 결합)</li>
<li>제한에 도달하면 전역 규칙이 우선</li>
</ul>
</li>
</ul>
<h3>규칙 모범 사례</h3>
<ul>
<li>글머리 기호 스타일로 형식 지정</li>
<li>XML 태그로 관련 규칙 그룹화</li>
<li>가장 중요한 규칙 우선시</li>
<li>코딩 표준 및 패턴에 대해 구체적으로 명시</li>
</ul>
<h2>프리뷰 설정</h2>
<ul>
<li><strong>설정</strong>: On/Off</li>
<li><strong>기능</strong>: 활성화되면 대화형 기능이 포함된 웹 애플리케이션의 IDE 내 미리보기를 제공합니다.</li>
</ul>
<p>다음을 통해 전환:</p>
<ul>
<li>설정 패널</li>
<li>도구를 호출하거나 명시적으로 비활성화하는 프롬프트</li>
</ul>
<p>프리뷰 기능:</p>
<ul>
<li>UI 요소를 Cascade로 전송</li>
<li>IDE 내 또는 별도의 브라우저에서 보기</li>
<li>대상 편집을 위한 요소 선택</li>
</ul>
<h2>고급 구성 옵션</h2>
<h3>Remote SSH 지원</h3>
<ul>
<li>원격 개발을 위한 내장 지원</li>
<li>참고: Microsoft의 "Remote - SSH" 확장 프로그램을 설치하지 마세요(Windsurf와 충돌)</li>
<li>로컬 dev 컨테이너 지원(원격 컨테이너는 지원하지 않음)</li>
</ul>
<h3>Gitignore 통합</h3>
<ul>
<li>인덱싱 및 검색을 위해 .gitignore 파일 준수</li>
<li>특정 사용 사례를 위한 gitignore 파일 액세스 옵션</li>
</ul>
<h3>로컬 인덱싱 구성</h3>
<ul>
<li>Windsurf가 코드베이스를 인덱싱하는 방법 구성</li>
<li>임베딩 생성 및 업데이트 빈도 설정</li>
<li>코드베이스 크기 최적화 옵션</li>
</ul>
